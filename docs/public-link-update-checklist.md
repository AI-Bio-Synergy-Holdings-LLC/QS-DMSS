# Public Link Update Checklist

Canonical website:

```text
https://qs-dmss.studio
```

Use this page when updating public surfaces that are not controlled directly by
the repository.

## GitHub

GitHub repository metadata should use:

- Homepage: `https://qs-dmss.studio`
- Description: `QS-DMSS Studio: evidence-first simulation lab for reproducible runs, evidence bundles, campaign comparison, and research-object exports.`

## PyPI

The next package release should expose the Studio site through project metadata:

- `Homepage`: `https://qs-dmss.studio`
- `QS-DMSS Studio`: `https://qs-dmss.studio`
- `Documentation`: `https://qs-dmss.studio`

Keep Repository, Issues, Releases, PyPI, DOI, and Reviewer links as supporting
project URLs.

## LinkedIn

Recommended Featured link:

```text
QS-DMSS Studio
https://qs-dmss.studio
```

Recommended short profile/update wording:

```text
QS-DMSS Studio is the canonical public front door for QS-DMSS, an Apache-2.0,
evidence-first simulation lab for reproducible package/evidence workflows. The
project is designed to make simulation runs inspectable as research objects:
configured, measured, bundled, verified, replayable, comparable, citable, and
ready for review.
```

Keep the claim boundary attached when space allows:

```text
QS-DMSS is beta software for reproducible package/evidence workflows, not
peer-reviewed scientific validation.
```

## Outreach Copy

When asking for review, lead with the website, then route technical reviewers
to the specific GitHub issue that matches their lane:

```text
QS-DMSS Studio is live at https://qs-dmss.studio.

QS-DMSS is beta software for reproducible package/evidence workflows, not
peer-reviewed scientific validation. If you are open to a small review, please
pick one lane: Fractal SSFM validation (#105), dry-run Slurm site-policy review
(#99), or builder/product roadmap feedback (#57).
```
